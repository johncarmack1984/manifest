# Packaged deploy

manifest's normal build needs Rust, cargo-lambda, Zig, Node, pnpm, just, and CDK on your machine. That toolchain is the real adoption wall. This directory packages it so you can stand the dashboard up **without installing any of it** — you bring only credentials (and, for now, a domain).

Two paths exist today, plus the roadmap to a true console one-click.

## Path A — one command, local Docker

Requires only **Docker** and **AWS credentials** on your machine. The pinned toolchain lives in [`Dockerfile`](Dockerfile); [`bootstrap.sh`](bootstrap.sh) builds it and runs the deploy inside it against your account.

```sh
cp infra/.env.example infra/.env   # then edit: domain, hosted zone id/name, Cognito prefix, owner email
export AWS_PROFILE=your-profile     # or export AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_SESSION_TOKEN
deploy/bootstrap.sh                 # builds the image, runs `cdk bootstrap`, then deploys
```

Notes:
- The repo is mounted into the container, so build outputs (`api/target`, `web/dist`, `node_modules`) land in your checkout as **Linux** artifacts. If you also build natively on this machine, use one path or the other — don't interleave — or `just` a clean between them.
- `--skip-bootstrap` skips the (idempotent) `cdk bootstrap` once the account/region is already bootstrapped.
- Only `us-east-1` is bootstrapped automatically. Extra `MANIFEST_INDEXED_REGIONS` each need their own `cdk bootstrap aws://<account>/<region>` (run it once per region).

## Path B — zero local toolchain, GitHub Codespaces

The [`.devcontainer`](../.devcontainer/devcontainer.json) points Codespaces (and VS Code Dev Containers) at the same image, so you get the whole toolchain in the browser with **nothing installed locally**.

1. On the repo in GitHub: **Code → Codespaces → Create codespace**.
2. In the codespace terminal: `cp infra/.env.example infra/.env` and edit it.
3. Authenticate AWS (`aws configure`, or `aws sso login`), then `just deploy`.

## Both paths still assume

- **A Route53 hosted zone you control** — the stack provisions an ACM cert + custom domain against it.
- **CDK bootstrap** in the target account (Path A does this for you).

These are the remaining friction after the toolchain is gone. See the roadmap below for removing them too.

## Roadmap — AWS console one-click (quick-create)

The end state is a **"Launch stack" button**: a CloudFormation [quick-create link](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/cfn-console-create-stacks-quick-create-links.html) that stands manifest up in a fresh account with no local anything. The engineering to get there, in order:

1. **Publish versioned artifacts.** A release workflow (on tag) builds the two arm64 Lambda zips + the SPA bundle and uploads them to a **public, regional S3 bucket** manifest owns (e.g. `manifest-artifacts-<region>`), keyed by version. This is the piece that lets CloudFormation fetch code without a local build or a per-account bootstrap bucket.
2. **A parameterized template, decoupled from CDK assets.** CDK's synthesized template references the account's bootstrap bucket, which quick-create can't satisfy. So the packaged template takes `ArtifactsBucket` + `Version` parameters and points Lambda `Code` / the SPA deployment at the public artifacts from step 1. It can be authored directly, or generated by overriding CDK's asset resolution — but it must not depend on `cdk bootstrap`.
3. **A no-domain mode.** Offer serving on the default CloudFront domain (`*.cloudfront.net`) so a first run needs **no Route53 zone or ACM cert** — the last local/account prerequisite. Custom domain stays available as a parameter for when you want it.
4. **The button.** README embeds `https://console.aws.amazon.com/cloudformation/home?region=us-east-1#/stacks/quickcreate?templateURL=<published-template-url>`, pre-filling the stack name and surfacing the same parameters as `infra/.env`.

Until then, Path A and Path B remove the toolchain wall, which is the largest part.
